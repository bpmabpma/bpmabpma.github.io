Implement i18n

My reference example is below.
Spring MVC Internationalization I18n Example
But, my example is always different because they are based on Spring Boot and Thymeleaf.
There are few things need to setup, e.g. MessageSource, LocaleResolver, and LocaleChangeInterceptor.
In Spring Boot, you don’t need to setup MessageSource Bean configuration. Spring Boot supports the property configuration.
It uses ReloadableResourceBundleMessageSource to implement to the message file loading.

application.properties

Then, we need to set up the LocaleResolver and the LocaleChangeInterceptor.
LocaleResolver has three type of class, e.g. CookieLocaleResolver, SessionLocaleResolver, and AcceptHeaderLocaleResolver.
It is used for determining the locale of the request based on which value source.
AcceptHeaderLocaleResolver should be the default choice, but I think CookieLocaleResolver is a better choice.
CookieLocaleResolver will check if there is no default locale and cookie setup, it will use the accept header value.
LocaleChangeInterceptor just enables and sets up the parameter name for changing the locale.

WebConfig.java

Then, if you access the page first time, you will setup the locale based on the accept header.
You can change the locale by using the lang parameter.
For example, if the URL is http://localhost:9000/emp/save?lang=zh, the locale will be changed to default.
If the URL is http://localhost:9000/emp/save?lang=zh, the locale will be changed to zh_TW.

Message Format

This topic relates to the validation. The validation also can setup the message format automatically.
Index 0 usually is filed name, you also can set up the field name on the message property file.
But, only the validation annotation can setup the parameters.

messages_zh_TW.properties

External Message properties

Check this StackOverflow article External Resource Properties.

  • classpath:messages - will always look up embeded message_xxx files
  • classpath:/messages and classpath:message are equivalent
  • file:messages - will look up jar’s external current directory e.g. ./message_en.properties <- this is what you want
  • file:/messages - you have to put your message files to root “/“ to make it work
  • use notation file:config/messages if you need to put in config folder together with your ./config/application.properties

I have counted some failure in this step. The configuration, file:messages, is based on the current directory of java -jar command.
So, I try to use the better way extract the message resource.

application-prod.properties

Then, I use the command to specify the spring profile.

But, there is another problem, I don’t want the message property files packed into the jar file.
An easy way to do it is to use Maven to exclude this file. And move the message property files in the target folder.

pom.xml

Package the jar, and run it.
